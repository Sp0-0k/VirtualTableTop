## 1. Overview

M5 lands fog of war. The DM gets a bottom dock with a round brush and a rectangle marquee, in Reveal or Hide mode, plus brush size and two bulk actions: Reset to fogged and Reveal everything. Strokes are vector-stored in the existing `fog_strokes` table (extended with a `shape` column) and rendered client-side by compositing into an offscreen canvas which is wrapped as a single `Konva.Image` layer.

Fog uses an **implicit base**: a page with zero strokes is fully fogged. Reveal strokes subtract from the fog; Hide strokes re-fog. Stroke ordering is insertion order. There is no special "page is fogged by default" flag and no giant initial stroke — empty list ⇒ black.

Real-time sync follows the parent spec's preview/commit pattern: `fog:stroke_preview` mid-drag (no DB), `fog:stroke_commit` on mouse-up (persisted + broadcast). Mid-stroke `fog:stroking` events go to **other DM tabs only**. Players see only post-commit `fog:stroke_added`. Bulk clear endpoints are HTTP REST.

**Done when:**
- DM paints reveals on the active page; players see the revealed area within ~200 ms of mouse-up.
- A new page is fully fogged for players (zero strokes ⇒ black).
- "Reset to fogged" returns the page to fully fogged. "Reveal everything" makes the whole page visible to players.
- Fog persists across reload (covered by the Playwright spec from parent §11).
- A stroke on a non-active page never reaches a player socket (filtered server-side).

## 2. Architectural decisions carried from earlier milestones

These three principles, set in M3 and M4, govern M5's shape:

1. **Config-frequency mutations are HTTP REST under `/api/dm/*`.** Bulk fog operations (clear all, reveal everything) are HTTP. They mutate many rows and benefit from idempotent resource semantics.
2. **Interaction-frequency mutations are Socket.IO commands.** Per-stroke painting uses `fog:stroke_preview` (ephemeral) and `fog:stroke_commit` (persisted), mirroring `token:move_*` from M4.
3. **Filter at the broadcast boundary.** Clients are never trusted to filter their own visibility. Fog broadcasts use the per-recipient builder pattern from M4: a player socket on a non-active page gets `null` and is skipped.

`state:full_sync` extends from M4's `{ activePage, tokens, players }` to include the active page's fog strokes. Non-active pages still ship without fog (DM private preview is M6).

## 3. Schema migration

`migrations/002_fog_stroke_shape.sql`:

```sql
ALTER TABLE fog_strokes
  ADD COLUMN shape TEXT NOT NULL DEFAULT 'brush'
    CHECK (shape IN ('brush', 'rect'));
```

Production rows: zero. The column default is `'brush'` so any latent rows would still satisfy the CHECK; in practice the table is empty when M5 ships.

The existing columns retain their meaning:

| `shape` | `points_json` | `radius` | meaning |
| --- | --- | --- | --- |
| `'brush'` | `[[x,y], …]` polyline (≥1 point, image-px coords) | brush radius in image px, > 0 | round-cap polyline; single-point click = filled disc |
| `'rect'` | `[[x1,y1],[x2,y2]]` two corners (normalized) | unused — store `0` | axis-aligned rectangle |

`mode` (`'reveal' | 'hide'`) is orthogonal to `shape`, giving four stroke variants.

## 4. Stroke representation & validation

**Stored shape:**
```ts
type FogStroke = {
  id: number;
  pageId: number;
  mode: 'reveal' | 'hide';
  shape: 'brush' | 'rect';
  points: [number, number][];   // image-pixel coords
  radius: number;               // > 0 for brush, 0 for rect
  createdAt: number;
};
```

**Server validation on commit:**
- `mode` ∈ `{'reveal', 'hide'}`.
- `shape` ∈ `{'brush', 'rect'}`.
- Points clipped to `[0, image_w] × [0, image_h]` of the page's background asset.
- `shape === 'brush'`: `points.length ≥ 1`, `radius` clamped to `[1, 4096]`.
- `shape === 'rect'`: `points.length === 2`, points normalized so `x1 ≤ x2, y1 ≤ y2`, zero-area rect rejected, `radius` forced to `0`.
- Page exists and the DM is authorized for it.

Invalid commits respond with an `error` event; no row created.

**Stroke ordering:** strokes are applied in `id` order (insertion order). A later hide over an earlier reveal re-fogs that area. The compositor reads strokes in `SELECT … ORDER BY id` order.

**Indexes:** existing `idx_fog_page` on `fog_strokes(page_id)` is sufficient. No new indexes.

## 5. HTTP endpoints

All under `/api/dm/*`, gated by Caddy `basic_auth` and a route-level `vtt_dm` cookie check (defense-in-depth, same pattern as M3/M4).

```
DELETE /api/dm/pages/:id/fog
  204 — delete every fog_strokes row for page
  Broadcasts fog:cleared { pageId } (filtered: players only on active page).
  404 if page not found.

POST /api/dm/pages/:id/fog/reveal-all
  204 — delete every row, then INSERT one
        shape='rect' mode='reveal' covering (0,0)..(image_w, image_h)
        of the page's background asset.
  Broadcasts fog:cleared { pageId } then fog:stroke_added (the new stroke),
  in that order, both filtered.
  409 if the page has no background_asset_id (no bbox to seed).
  404 if page not found.
```

Both endpoints are guarded client-side by a confirm dialog.

## 6. Socket events

### Client → server (commands)

```
fog:stroke_preview   DM only.  { pageId, mode, shape, points, radius }
                     Throttled ~30 Hz client-side. No DB. No validation
                     beyond role + page exists. Broadcast as fog:stroking
                     to OTHER DM SOCKETS ONLY. Players never see this event.

fog:stroke_commit    DM only.  { pageId, mode, shape, points, radius }
                     Validates per §4. INSERT one row. Broadcasts
                     fog:stroke_added (full row including id) to:
                       - all DM sockets
                       - all player sockets currently on the active page
                         (i.e., only if pageId === activePageId).
```

A player socket emitting either command receives an `error` event and the command is dropped.

### Server → client (events)

```
fog:stroking         { pageId, mode, shape, points, radius }
                     DM tabs only. Ephemeral; never persisted; never re-sent on
                     reconnect.

fog:stroke_added     { stroke: FogStroke }
                     DM tabs always; player sockets only when
                     stroke.pageId === activePageId.

fog:cleared          { pageId }
                     DM tabs always; player sockets only when
                     pageId === activePageId.
```

### `state:full_sync` extension

```ts
{
  activePage: {
    id: number,
    name: string,
    backgroundAsset: AssetRef | null,
    gridWidthSquares: number,
    gridHeightSquares: number,
    strokes: FogStroke[],          // NEW in M5; active page only
  } | null,
  tokens: Token[],                  // unchanged from M4
  players: PlayerSummary[],         // unchanged
}
```

Non-active pages still ship without fog. DM private preview (M6) will extend this.

### Filtering invariant

Outbound `fog:*` events use the per-recipient builder pattern from M4:

```ts
function fogPayloadFor(socket, payload, activePageId) {
  if (socket.data.role === 'dm') return payload;
  if (payload.pageId === activePageId) return payload;
  return null;
}
```

Skip-on-`null` keeps the active-page filter centralized.

## 7. Client rendering

### Compositor module — `client/src/canvas/fogBitmap.ts`

A pure function `buildFogBitmap(w, h, strokes)` that returns an `HTMLCanvasElement`:

```ts
export function buildFogBitmap(
  w: number, h: number, strokes: FogStroke[]
): HTMLCanvasElement {
  const c = document.createElement('canvas');
  c.width = w; c.height = h;
  const ctx = c.getContext('2d')!;
  ctx.fillStyle = '#000';
  ctx.fillRect(0, 0, w, h);                    // implicit base: fully fogged
  for (const s of strokes) {
    ctx.globalCompositeOperation =
      s.mode === 'reveal' ? 'destination-out' : 'source-over';
    ctx.fillStyle = '#000';
    if (s.shape === 'rect') {
      const [[x1,y1],[x2,y2]] = s.points;
      ctx.fillRect(x1, y1, x2 - x1, y2 - y1);
    } else {
      // brush
      ctx.lineWidth = s.radius * 2;
      ctx.lineCap = 'round';
      ctx.lineJoin = 'round';
      ctx.strokeStyle = '#000';
      if (s.points.length === 1) {
        const [[x,y]] = s.points;
        ctx.beginPath();
        ctx.arc(x, y, s.radius, 0, Math.PI * 2);
        ctx.fill();
      } else {
        ctx.beginPath();
        ctx.moveTo(s.points[0][0], s.points[0][1]);
        for (let i = 1; i < s.points.length; i++) {
          ctx.lineTo(s.points[i][0], s.points[i][1]);
        }
        ctx.stroke();
      }
    }
  }
  return c;
}
```

Pure function with no Konva or React dependency — directly testable in vitest + jsdom.

### Konva integration — `client/src/canvas/FogLayer.tsx`

```tsx
<Layer opacity={role === 'dm' ? 0.5 : 1.0} listening={false}>
  <Image image={fogBitmap} x={0} y={0} width={imageW} height={imageH} />
</Layer>
```

- Subscribes to `(activePage.strokes, dmInProgressStroke)` from the Zustand store.
- Rebuilds the bitmap whenever either changes (single `useMemo` over the concatenated stroke array).
- `listening={false}` so the layer never intercepts clicks/drags; the token layer above keeps full pointer behavior.

### Layer order (z bottom → top)

1. Background map (Konva.Image, the page's background asset)
2. Grid overlay (M4)
3. Tokens (M4)
4. Fog (M5)
5. Selection ring + popover overlay (M4) — DM-only, deliberately above fog so the DM can still see selection on faintly-fogged areas

### In-progress stroke

While the DM is dragging, the partial stroke lives in the store as `dmInProgressStroke: FogStroke | null`. The compositor passes `[...persistedStrokes, dmInProgressStroke].filter(Boolean)` so the DM sees their own paint live.

On mouse-up:
1. Client emits `fog:stroke_commit`.
2. Server persists, broadcasts `fog:stroke_added`.
3. Client appends the canonical stroke to `persistedStrokes` and clears `dmInProgressStroke`.

Order is preserved: in-progress sat at end while painting; canonical sits at end after commit.

### Recompute cost

A 4096 × 4096 canvas with a few hundred strokes rebuilds in < 5 ms in Chrome. No incremental dirty-tracking needed at this scale; the entire bitmap is rebuilt on every change.

### Player-side optimization

Players never receive non-active-page strokes (server filter), so `buildFogBitmap` only ever runs for the active page on a player client.

## 8. DM dock UI

When the **Fog** tool is selected in the existing top toolbar (next to Select), a horizontal dock appears at the bottom of the canvas. Switching back to Select hides the dock and clears any in-progress stroke.

### Dock layout (left → right)

```
┌──────────────────────────────────────────────────────────────────────────┐
│  [Reveal] [Hide]   │   [Brush] [Rect]   │   Size ─────●────  120px       │
│                                                                          │
│                                              [Reset to fogged]           │
│                                              [Reveal everything]         │
└──────────────────────────────────────────────────────────────────────────┘
```

- **Mode group (Reveal | Hide):** segmented control. Default Reveal.
- **Shape group (Brush | Rect):** segmented control. Default Brush.
- **Size slider:** brush radius 8 – 400 image px. Disabled when shape = Rect. Live numeric label.
- **Reset to fogged:** confirm dialog ("This will re-fog the entire page. Continue?") → `DELETE /api/dm/pages/:id/fog`.
- **Reveal everything:** confirm dialog ("This will reveal the entire page to players. Continue?") → `POST /api/dm/pages/:id/fog/reveal-all`.

### Canvas interaction while Fog tool is active

- Pan and zoom keep working (drag-empty-canvas pan, wheel zoom — same as M4).
- Pointer-down on the canvas starts a stroke. Cursor shows a circle preview at brush radius (Brush mode) or a crosshair (Rect mode).
- **Token layer is set to `listening={false}`** while the Fog tool is active, so the DM cannot accidentally drag a token while painting. Selection state is preserved; switching back to Select restores listening.

#### Brush mode

1. mousedown → push first point into the in-progress stroke; start emitting `fog:stroke_preview` at 30 Hz.
2. mousemove → push points (decimated: skip if within 2 image-px of the last point).
3. mouseup → emit `fog:stroke_commit`; clear in-progress.

#### Rect mode

1. mousedown → record corner A.
2. mousemove → second corner is the current cursor position; emit throttled `fog:stroke_preview` with `points = [A, current]`.
3. mouseup → emit `fog:stroke_commit` with normalized `[A, B]`. If `A === B` (zero-area), discard with no commit.

#### Abort

Right-click or Esc mid-stroke aborts: clear in-progress, no commit, no broadcast.

### Players don't see the dock

The Fog tool toggle and dock are DM-only. The player view is unchanged from M4 except for the new fog layer rendering.

## 9. Testing

Three layers, scoped to fog-specific behavior. Existing M3/M4 tests already cover the broader sync and auth surface.

### Server unit (vitest)

- `migrations/002_fog_stroke_shape.sql` applies idempotently and adds the `shape` column with default `'brush'`.
- Stroke validator: rejects brush with 0 points, rect with ≠ 2 points, brush radius < 1 or > 4096, unknown mode/shape, zero-area rect.
- Stroke clipping: points outside `[0, w] × [0, h]` are clipped; rect normalized so `x1 ≤ x2, y1 ≤ y2`.
- Per-recipient filter `fogPayloadFor(socket, payload, activePageId)`:
  - DM always receives.
  - Player receives when `payload.pageId === activePageId`.
  - Player + non-active-page payload ⇒ `null`.

### Server integration (vitest + supertest + Socket.IO test harness)

- `state:full_sync` for a player includes `activePage.strokes`; non-active pages have no `strokes` field (or it's empty).
- DM `fog:stroke_commit` for the active page broadcasts `fog:stroke_added` to a connected player socket.
- DM `fog:stroke_commit` for a **non-active** page broadcasts to other DM tabs but **not** to the player.
- `DELETE /api/dm/pages/:id/fog` deletes all rows and broadcasts `fog:cleared`.
- `POST /api/dm/pages/:id/fog/reveal-all` deletes rows, inserts one bbox-rect reveal stroke, broadcasts `fog:cleared` then `fog:stroke_added` in that order.
- A player socket emitting `fog:stroke_commit` receives an `error` event; no row created.

### Client unit (vitest, jsdom)

- `buildFogBitmap` snapshot tests against fixture stroke lists:
  - Empty (fully black bitmap).
  - Single reveal disc (transparent disc on black background).
  - Reveal then hide at the same location (re-fogged).
  - Rect reveal.
- Stroke decimation: feeding the brush handler 1000 close-together pointermoves yields a polyline shorter than 1000 points.

### Browser smoke (Playwright)

One new spec extending parent §11:

`fog-roundtrip.spec.ts` — DM logs in, opens an active page, switches to Fog tool, paints a reveal stroke. Player browser sees a reveal in the same area within 200 ms. DM reloads — fog persists. DM hits Reset to fogged — player sees fully black.

### Explicitly not tested

- Pixel-perfect fog edges (DM and player composite identically by construction since `buildFogBitmap` is shared).
- Slider micro-interactions.
- Confirm-dialog wording.

## 10. Out of scope (carried to M6 or later)

- DM private preview of fog on non-active pages — M6.
- Dynamic line-of-sight, walls, per-token vision — M7 (parent spec stretch).
- Light radius rendering — M7+.
- Fog stroke editing/undo (no remove-individual-stroke UI; the DM re-paints with Hide if they over-revealed).
- Soft-edge brushes / opacity blending — round hard-edge brush only.
