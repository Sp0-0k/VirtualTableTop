# M3 — Pages, Maps, Asset Upload — Design Spec

**Date:** 2026-05-07
**Status:** Approved for planning
**Owner:** Kirk
**Parent spec:** `docs/superpowers/specs/2026-04-27-vtt-design.md`

## 1. Overview

M3 stands up the asset-upload pipeline (maps only), pages CRUD, the DM library/sidebar UI, and a player view that renders the active page's background image. No tokens, no grid overlay, no fog yet — those land in M4 / M5.

**Done when:**

- DM uploads a map image, creates a page using it, sets the page active.
- Player sees that map full-bleed (read-only `<img>`, `object-fit: contain`).
- DM can create / select / delete pages from the sidebar; "Set Active" promotes one to all players.
- New uploads dedup by content hash; identical re-upload returns the existing row.
- All existing M2 tests still pass; new tests cover the upload pipeline, pages CRUD, and active-page broadcast.

The M3 schema is already present in `migrations/001_initial.sql` (assets, pages, tokens, fog, walls were created at M2 time per the parent spec). M3 adds **no migrations** — it's pure server + client code.

## 2. Architectural decision: HTTP for config-like CRUD

The parent spec §6 lists `page:create / page:update / page:delete / page:set_active` as Socket.IO client→server commands. In M3 these become **HTTP REST endpoints under `/api/dm/*`** instead. Asset upload is HTTP regardless (multipart). Sockets remain reserved for high-frequency interaction events that arrive in later milestones (`token:move_*` in M4, `fog:stroke_*` in M5).

The broadcast events (`page:created`, `page:updated`, `page:deleted`, `state:active_page_changed`, `state:full_sync`, `asset:created`) remain Socket.IO server→client events as the parent spec defines.

**Why:**

- Asset upload must be HTTP (multipart). Splitting page CRUD over a different transport is gratuitous.
- HTTP is debuggable with curl, has standard status codes, and matches the existing M2 pattern (`/api/player/join`, `/api/me`).
- Page CRUD is click-frequency, not interaction-frequency — there is no latency benefit to socket commands.
- The split between "config-like mutations" (HTTP) and "interaction-like mutations" (sockets) is meaningful, not arbitrary.

The parent spec §6 will be patched in the same commit as this spec to reflect the change.

## 4. HTTP endpoints

All under `/api/dm/*`, all DM-only (Caddy `basic_auth` per M2; routes also re-check the `vtt_dm` cookie so dev mode without Caddy is still safe).

```
POST   /api/dm/assets/upload    multipart: file (max 5 MB), kind=map
                                201 { id, hash, kind, original_name, mime, width, height, size_bytes }
                                200 same payload on dedup hit
                                400 bad mime / oversize dimensions
                                413 payload too large (>5 MB)
                                429 rate limited (10/min per session)
                                507 disk quota exceeded (>5 GB total)

GET    /api/dm/assets?kind=map  200 [{ asset, … }] sorted by uploaded_at desc

GET    /api/dm/pages            200 [{ page, … }] sorted by sort_order asc

POST   /api/dm/pages            body: { name, background_asset_id, grid_width_squares, grid_height_squares }
                                201 page (sort_order = current max + 1, is_active = 0)
                                400 invalid body / unknown asset_id

PATCH  /api/dm/pages/:id        body: subset of { name, background_asset_id,
                                                  grid_width_squares, grid_height_squares, sort_order }
                                200 page; 404 unknown id

DELETE /api/dm/pages/:id        204 on success
                                409 if page is active (must set another active first)
                                404 unknown id

PUT    /api/dm/pages/:id/set-active   200 page (atomic TX)
                                       404 unknown id
```

Player-facing static asset URLs:

```
GET /assets/<hash>.webp           Caddy in prod, Express in dev
GET /assets/<hash>.thumb.webp     Caddy in prod, Express in dev
```

`/assets/*` is *not* gated by Caddy basic_auth, on the rationale that hash URLs are unguessable. Players still have to clear `/`-realm basic_auth to reach the app at all.

**Deferred to a later milestone:** `DELETE /api/dm/assets/:id`. Cruft accumulates but is harmless at one-campaign scale; defining the reference-checking semantics is not worth M3 scope.

## 5. Socket.IO events (M3 surface)

**Server → DM-only (`io.to('dm')`):**

```
asset:created     { asset }                 // after successful new upload
page:created      { page }
page:updated      { page }
page:deleted      { id }
```

**Server → all (DM + players):**

```
state:full_sync             { activePage: Page | null }   // emitted once per connection on connect
state:active_page_changed   { activePage: Page | null }   // after PUT set-active or active-page DELETE
```

**Client → server:** *none in M3.* All commands are HTTP. Sockets are receive-only from the client side until M4.

The DM-only events are filtered by joining DM sockets to a `dm` room at the end of the auth middleware (`if (socket.data.role === 'dm') socket.join('dm')`), and emitting via `io.to('dm').emit(...)`. Players never join the room and therefore never receive these events. This matches the parent spec's "filter at the broadcast boundary" rule (§6) without needing a per-socket loop yet.

### `Page` payload shape (broadcast version)

```ts
{
  id: number,
  name: string,
  background_asset_id: number | null,
  background_url: string | null,         // server-resolved: '/assets/<hash>.webp' or null
  grid_width_squares: number,
  grid_height_squares: number,
  sort_order: number,
  is_active: 0 | 1,
}
```

The server resolves `background_url` server-side so clients never join asset rows themselves. Saves a round-trip and keeps client state dumb. (Same payload shape is returned by `GET /api/dm/pages` and `POST /api/dm/pages`.)

### `state:full_sync` is forward-extensible

In M3 the payload is `{ activePage }`. Later milestones extend it: M4 adds `tokens` and `players`, M5 adds `fog`. The parent spec §6 lists this event with its eventual full shape; M3 ships a strict subset.

## 6. Image pipeline

`server/src/assets/pipeline.ts` is a pure-ish module: it takes a buffer + kind and returns `{ hash, processedBytes, thumbBytes, metadata }` without touching disk or DB. The route handler orchestrates I/O:

```
1. multer single-file parse, 5 MB hard cap → buffer in memory
2. file-type from buffer → must be image/png | image/jpeg | image/webp; else 400
3. sharp(buffer).metadata() → reject if width or height > 8192 px
4. sharp resize:
     maps:   max 4096 px longest edge, fit 'inside', WebP q85
5. sha256(processed bytes) → hash
6. SELECT assets WHERE hash = ?
     hit: return existing row (200, no file write, no broadcast)
     miss: continue
7. sharp thumbnail: 256×256 cover, WebP → buffer
8. Atomic write: temp file → fsync → rename to <hash>.webp and <hash>.thumb.webp
9. INSERT assets row
10. io.to('dm').emit('asset:created', { asset })
11. 201 { asset record }
```

**Edge cases:**

- *Concurrent identical uploads:* both compute the same hash; second `INSERT` violates `UNIQUE(hash)`; we recover by re-`SELECT` and return that row.
- *Crash between file write and DB INSERT:* leaves an orphan `.webp` on disk. Acceptable in v1 — periodic GC sweep is post-MVP polish.
- *Crash between DB INSERT and file write:* impossible by ordering — file write happens before INSERT.

**New deps:** `sharp`, `multer`, `file-type`. All standard, well-maintained.

DM Zustand store (new in M3):

```ts
{
  pages: Page[],
  selectedPageId: number | null,   // DM private preview, NOT players' active page
  activePageId: number | null,
  assets: Asset[],                 // maps only in M3
}
```

DM interactions:

- Click a page in the sidebar → `selectedPageId = id` (private preview, no network).
- "Set Active" button on the selected page → `PUT /api/dm/pages/:id/set-active`.
- "+ New page" → modal: name input, map dropdown (from `assets`), default 20×15 grid → `POST /api/dm/pages`.
- "+ Upload" → file picker → `POST /api/dm/assets/upload` (multipart).
- "×" on a page (with confirm) → `DELETE /api/dm/pages/:id`. Toast on 409 ("can't delete the active page").

## 9. Testing strategy

### Unit (vitest, in-memory SQLite, tmpdir)

- `pipeline.ts`: PNG fixture → small WebP, dimensions match expected resize, output mime is webp.
- `pipeline.ts`: oversize fixture (>8192 px) rejects.
- `pipeline.ts`: corrupt-bytes fixture rejects.
- `pipeline.ts`: same fixture twice → identical hash.
- `db/pages.ts`: `setActive` is exclusive (calling twice with different ids leaves only one is_active=1).
- `db/pages.ts`: delete refuses when row is active.
- `db/assets.ts`: dedup hit returns existing row.

### Integration (supertest)

- `POST /api/dm/assets/upload` happy path → 201, row inserted, file on disk.
- Same upload twice → second returns 200 with same id, no new file.
- Bad mime (text/plain) → 400.
- Oversize body (>5 MB) → 413.
- `GET /api/dm/assets?kind=map` lists.
- `POST /api/dm/pages` happy path → 201, sort_order = max+1.
- `PUT /api/dm/pages/:id/set-active` → exactly one page is_active=1 afterward.
- `DELETE /api/dm/pages/:id` on active page → 409.
- Any `/api/dm/*` without `vtt_dm` cookie → 401 (route-level self-check).

### Socket integration (existing harness)

- DM connects → receives `state:full_sync { activePage }` (matches DB).
- DM sets active via HTTP → DM and player sockets both receive `state:active_page_changed` with resolved `background_url`.
- Player connects mid-session → `state:full_sync` includes current activePage.
- DM creates a page → DM socket receives `page:created`; player socket does NOT.

### Skipped for M3

- Playwright (UI is too thin to drive end-to-end; pick up in M4).
- Performance / load tests.

## 10. Deferred / explicitly out of scope

- Asset deletion (`DELETE /api/dm/assets/:id`) — defer to a later milestone; cruft is harmless at this scale.
- Page reorder UI — `sort_order` is settable via PATCH but no drag-reorder UI yet.
- Token uploads — M4 territory.
- Grid overlay rendering — M4.
- Fog of war — M5.
- Konva canvas — M4.
